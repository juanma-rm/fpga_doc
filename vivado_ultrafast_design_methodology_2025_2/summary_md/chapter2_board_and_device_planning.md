# Chapter 2: Board and Device Planning

Planning device orientation, signal-to-pin assignment, and PCB layout leads to dramatic improvements in system performance, power consumption, thermal performance, and design cycle times. This chapter covers PCB layout, power distribution, clock planning, I/O assignment, SSI device design, HBM device design, and configuration.

## PCB Layout Recommendations

### Aligning with Physical Components on the PCB

- Orient the device so **GT interfaces** are as close as possible to the PCB components they connect to
- Use a sketch of the PCB with critical interfaces to determine optimal device orientation
- For high-speed interfaces (memory), prefer **package pins closest to the device edge** — short, direct, matched-length traces without vias
- Use the Vivado IDE **I/O Planning view layout** for visualization

> **Thermal Tip:** Avoid placement where the device is in the exhaust of another high-power component. Run thermal simulation to understand placement impact on junction temperature.

### Power Distribution System (PDS)

Key factors for PDS design:

| Factor | Recommendation |
|--------|---------------|
| **Voltage regulators** | Select based on noise and current requirements from power estimation |
| **Shunt resistors** | Add to monitor power on each rail (or use PMBus-enabled regulators) |
| **Sysmon supply** | Set up VCCAUX_SMON |
| **PDN simulation** | Verify decoupling capacitor count; run SPICE/SIMPLIS simulation |
| **Power consolidation** | See UG583 for UltraScale consolidation options |

Power estimation tools:
- **XPE** (Xilinx Power Estimator) — for 7 series and UltraScale devices
- **PDM** (Power Design Manager) — for UltraScale+ devices

> **Power Tip:** Use Vivado `report_power` to analyze power on a per-regulator or per-VRM basis.

### Thermal Solution Considerations

| Parameter | Description |
|-----------|-------------|
| **Junction Temperature Tj** | Override to match thermal simulation; set to worst case if no simulation |
| **Effective ΘJA** | Thermal efficiency in °C/W (e.g., 2.1°C/W → 10W design = 21°C above ambient) |

Formula: $\Theta_{JA} = \frac{T_j - T_a}{P_{dissipated}}$

> **Thermal Tip:** Use lidless packaging when available — more efficient thermal solution with direct heat source contact. Apply 20–50 PSI heat sink pressure with 4-hole mounting.

```tcl
# XDC operating condition constraints (export from XPE/PDM)
set_operating_conditions -process Maximum
set_operating_conditions -design_power_budget <value>

# If thermal simulation completed:
set_operating_conditions -ambient_temp <value>
set_operating_conditions -thetaja <value>

# Else if no thermal simulation:
set_operating_conditions -junction_temp <value>
```

### PCB Design Steps

1. Review PCB Design Guide and transceiver user guide for your device
2. Review memory IP and PCIe design guidelines in IP product guides
3. Validate I/O planning in Vivado: run SSN analysis, built-in DRCs, export IBIS models
4. Run signal integrity analysis (Spice/IBIS-AMI for GTs, IBIS for lower-speed)
5. Use XPE/PDM with **Process = Maximum** for early power estimate
6. Complete schematic checklist for your device
7. Import XDC operating conditions file generated by XPE/PDM

## Device Power Aspects and System Dependencies

### Power Types

| Type | Description |
|------|-------------|
| **Power-On** | Transient spike current at first power application — not a concern with proper sequencing |
| **Startup** | Power during initial bring-up and configuration — short duration, but must meet current requirements |
| **Static (Standby)** | Minimum continuous power when configured with no activity — function of junction temperature |
| **Dynamic** | Power during operation with switching activity — based on average toggling over time |

> **Total Power = Static Power + Dynamic Power**

### Factors Impacting Power

| Factor | Impact |
|--------|--------|
| **Environmental** | Voltage and junction temperature affect power dissipation |
| **Power rail consolidation** | Discrete rails enable power domain switching at cost of more regulators |
| **Power model accuracy** | Evolves with device availability and manufacturing maturity |

> **Power Tip:** Compare total power to budget during design process:
> ```tcl
> set_operating_conditions -design_power_budget <Power in Watts>
> ```

## Clock Resource Planning and Assignment

Select clocking resources as **one of the first design steps**, well before pinout selection:

- Clocking selections can dictate pinout and direct logic placement
- Especially critical for SSI technology devices
- Consider constraint creation with clock planning, particularly in large/high-utilization devices
- Manual placement of clocking resources may be needed for design closure

> **See Also:** [Chapter 3: Design Creation with RTL — Clocking Guidelines](chapter3_design_creation_with_rtl.md#clocking-guidelines)

## I/O Planning Design Flows

The Vivado IDE provides interactive I/O exploration, visualization, assignment, and validation with **correct-by-construction** guarantees.

### Project Types for I/O Planning

| Type | Capabilities |
|------|-------------|
| **I/O Planning Project** | Easy entry point; specify I/O constraints; generate top-level RTL from defined pins |
| **RTL Project** | Synthesis + implementation; more comprehensive DRCs; IP core generation for memory/GT pinout planning |

> **Tip:** Start with an I/O planning project and migrate to an RTL project later.

### Pre-RTL I/O Planning

- Import I/O definitions via CSV or XDC file, or create dummy RTL with port directions
- Port direction availability improves SSN analysis accuracy
- Basic I/O bank DRC rules available

### Netlist-Based I/O Planning (Recommended)

- Assign I/Os after synthesis when clock logic paths are established
- More comprehensive DRC coverage

### Pin Assignment Guidelines

| Guideline | Detail |
|-----------|--------|
| **Group same-interface signals** | Place data, address, control pins in same bank (or adjacent banks in same SLR for SSI) |
| **Center control signals** | Place clocking, enables, resets, strobes in the middle of data buses |
| **Center high-fanout signals** | Place design-wide control signals toward device center (middle SLR for SSI) |

### Pin-Compatible Devices

Identify alternate devices to ensure I/O pin compatibility across device sizes (same package required).

> **Tip:** Plan device selection, pinout, and design criteria carefully at the start. Consider pinout, clocking, and resource management when migrating to larger/smaller devices.

### Specific I/O Types

| Interface | Considerations |
|-----------|---------------|
| **Memory** | Group IP ports into I/O Port Interface; use Memory Bank/Byte Planner; avoid bunching near center I/O column; follow trace length matching |
| **Gigabit Transceivers (GT)** | Consider reference clock sharing, PLL sharing within quad, hard block proximity, SLR boundary crossing; use GT wizard |
| **High Speed I/O** | Choose between HP (high-performance) and HR (high-range) banks based on speed |
| **Internal VREF / DCI Cascade** | Can free up pins for regular I/O; ensure related DRCs are run |

### Interface Bandwidth Validation

Create small connectivity designs to validate each interface:
- Full DRC checks on pinout, clocking, timing
- Hardware test design when board returns
- Rapid implementation for fastest interface debug
- Options: IBERT for SerDes, IP example designs, AXI traffic generators

## Designing with SSI Devices

### SSI Architecture

| Component | Description |
|-----------|-------------|
| **SLR (Super Logic Region)** | Single device die slice containing CLBs, block RAM, DSP, GTs |
| **Silicon Interposer** | Passive layer routing configuration, global clocking, general interconnect between SLRs |
| **SLL (Super Long Line)** | Routes connecting signals between SLRs — the only data connections between SLRs |
| **Master SLR** | Contains primary configuration logic, DNA_PORT, EFUSE_USER |

```tcl
# Query SLR information
llength [get_slrs]                                  ;# Number of SLRs
get_slrs -of_objects [get_cells my_cell]            ;# SLR of a cell
get_slrs -filter IS_MASTER                          ;# Master SLR
get_property NUM_TOP_SLLS [get_slrs SLR0]           ;# Available SLLs
```

### SSI Pinout Rules

- Group all pins of a particular interface into the **same SLR**
- For signals driving multiple SLRs, place in the **middle SLR**
- Balance CCIO/CMT components across SLRs
- Reduce SLR crossings

### Propagation Limitations

The following do **NOT propagate** across SLR boundaries:
- Carry chains
- DSP cascades
- Block RAM and UltraRAM cascades
- Other dedicated connections (DCI cascades)

### SLR Utilization

- Use Pblocks to assign logic to each SLR: `resize_pblock pblock_SLR0 -add SLR0`
- Validate that individual SLRs don't have excessive utilization
- Keep tightly interconnected modules within the same SLR
- Add pipeline registers at module interfaces for SLR crossing flexibility

> **Example:** 70% block RAM utilization overall may cause issues if one SLR exceeds 85%.

### SLR Crossing for Wide Buses

For buses >250 MHz crossing SLRs, use **at least 3 pipeline stages**: top, middle, and bottom of the SLR.

> **Tip:** Use AXI Register Slice IP or custom auto-pipelining IP for wide bus SLR crossings.

## Designing with HBM Devices

### HBM Architecture

Virtex UltraScale+ HBM devices contain:
- One or two 4 GB HBM stacks (up to 8 GB total)
- 32 HBM AXI interfaces
- Built-in switch for flexible addressing (any AXI interface → any memory address)
- HBM stacks connected through silicon interposer at device bottom

### Placement Considerations

| Guideline | Detail |
|-----------|--------|
| **SLR2 to HBM paths** | Often require 5+ pipeline stages; keep vertically aligned |
| **Lower fabric utilization in SLR0** | Budget for HBM AXI interface resources |
| **MIG IP in SLR0** | May cause timing challenges — consider SLR1/SLR2 I/O columns |
| **HBM AXI physical location** | Use interfaces on same device side as addressed HBM stack for optimal latency/bandwidth |
| **PCIE4C in SLR0** | Use sites farthest from HBM AXI interfaces (PCIE4CE4_X0Y1, PCIE4CE4_X1Y1) |

> **Tip:** Use auto-pipelining (AXI Register Slice IP) for 450 MHz timing closure between HBM interfaces and any SLR.

## Configuration

Configuration loads application-specific data into the device's volatile CMOS configuration latches. Data must be reloaded at each power-up.

### Configuration Sources

- External non-volatile memory (self-loading)
- External smart sources: microprocessor, microcontroller, DSP, PC, board tester

### Board Design Tips

| Signal | Recommendation |
|--------|---------------|
| **JTAG** | Always accessible, even when not primary config mode — enables ID check, DNA info, status registers, indirect flash programming |
| **INIT_B** | Connect to LED via driver with pull-up — indicates initialization completion and CRC errors |
| **DONE** | Connect to LED via driver with pull-up |

> **Recommended:** Choose configuration mode matching system requirements. Consider dedicated vs. dual-purpose pins, voltage restrictions on I/O banks, and suitable terminations.

## Best Practices

1. **Plan I/O in conjunction with board layout** — failing to do so decreases performance and extends design closure
2. **Select clocking resources first** — before pinout selection; clocking dictates I/O and logic placement
3. **Design PDS for worst-case power** with Process = Maximum in XPE/PDM
4. **Run thermal simulation** before finalizing thermal solution — use exported XDC operating conditions
5. **Register signals crossing SLR boundaries** — use 3+ pipeline stages for buses >250 MHz
6. **Group interface signals in the same SLR** and place control signals centrally
7. **Validate interfaces with small connectivity designs** before full design implementation
8. **Ensure JTAG, INIT_B, and DONE signals are accessible** for board debug
9. **Balance resource utilization across SLRs** — check with utilization reports; use SLR Pblocks
